# Supabase CDC end-to-end test setup

`TestAccSupabaseCDCIngestion_endToEndData` provisions a Fargate worker that
streams CDC events from a Supabase Postgres publication into Rawtree, then
inserts a handful of rows directly into the source table and waits for them
to appear in Rawtree. To keep the test simple the schema is **fixed** — it
expects the canonical "Superstore Sales" dataset (the CSV in this folder)
imported into a specific table name with specific column names.

This document walks through the one-time setup, then how to run the test.

The test assumes:

| Item | Value |
|---|---|
| Source schema | `public` |
| Source table | `superstore_sales_data` |
| Publication | `rawtree_superstore_sales_data_publication` |
| Primary key | `"Row ID"` (BIGINT) |
| Rawtree destination table | `public_superstore__sales__data` (created automatically by the worker) |

If you deviate from any of these names, edit the constants at the top of
[resource_acc_test.go](../../internal/resources/supabase_cdc_ingestion/resource_acc_test.go).

## 1. Create a Supabase project

Sign in at [supabase.com](https://supabase.com), create a new project, and
wait for it to provision. Note the **project ref** (the random alphanumeric in
URLs like `https://app.supabase.com/project/<ref>`); you'll use it below.

## 2. Import the dataset (creates the table)

The dataset is `dataset.zip` in this folder — unzip it to get `train.csv`
(~9.8k rows of the classic Tableau Superstore dataset):

```sh
unzip dataset.zip
```

In the Supabase dashboard, open **Table Editor → New table → Import data
from CSV**, pick `train.csv`, and name the table `superstore_sales_data`
(schema `public`). Supabase will:

- Create the table for you — no `CREATE TABLE` needed.
- Preserve the CSV column names verbatim (including spaces, e.g. `"Row ID"`,
  `"Order ID"`, `"Sub-Category"`). The test's INSERT statement uses these
  exact names, so don't rename them.
- Add its own `id bigint generated by default as identity primary key` column.
  This is fine — the test doesn't touch `id`; PG auto-populates it.
- Infer column types from the data. The dataset uses `DD/MM/YYYY` for dates,
  which Postgres can't parse into `DATE`, so accept whatever Supabase
  suggests (typically `text`). The worker passes values through as strings
  to Rawtree regardless of column type.

When the import finishes you should see ~9994 rows.

> **Alternative — `psql \copy`:** if you have psql installed locally and
> network access to the direct database, and you've created an empty table
> with matching columns first:
>
> ```sh
> psql "$RAWTREE_TEST_SUPABASE_DATABASE_URL" -c "\copy public.superstore_sales_data FROM 'train.csv' WITH (FORMAT csv, HEADER true)"
> ```

## 3. Enable logical replication for the table

Open **SQL Editor** and run:

```sql
-- REPLICA IDENTITY FULL makes UPDATEs / DELETEs include the full old row,
-- which the supabase/etl worker needs to emit complete CDC events.
ALTER TABLE public.superstore_sales_data REPLICA IDENTITY FULL;

-- Drop any prior publication for this table (idempotent setup).
DROP PUBLICATION IF EXISTS rawtree_superstore_sales_data_publication;

CREATE PUBLICATION rawtree_superstore_sales_data_publication
FOR TABLE public.superstore_sales_data;
```

## 4. Download the Supabase CA certificate

Supabase's Postgres certificate isn't signed by a CA in the standard Mozilla
bundle, so the worker will reject the TLS handshake unless we hand it the CA
explicitly. In the Supabase dashboard, open **Project Settings → Database →
SSL Configuration** and click **Download certificate**. Save it somewhere
your shell can reach — for the rest of this doc we assume
`~/supabase-ca.crt`.

You can also dump it from the live handshake if the dashboard download isn't
handy:

```sh
host=db.<your-project-ref>.supabase.co
openssl s_client -showcerts -starttls postgres -connect "$host:5432" </dev/null 2>/dev/null \
  | awk '/-----BEGIN CERT/,/-----END CERT/' > ~/supabase-ca.crt
```

## 5. Get the connection URLs

The worker and the test process need different URLs because Supabase's direct
endpoint is IPv6-only (typically) and most laptops have no IPv6 route.

In the Supabase dashboard, open **Project Settings → Database → Connection
string**. You'll see several variants — copy both:

| Use | UI label | Looks like |
|---|---|---|
| Worker (logical replication) | **Direct connection** | `postgres://postgres:<pw>@db.<ref>.supabase.co:5432/postgres?sslmode=require` |
| Test process (plain INSERTs from your laptop) | **Session pooler** | `postgres://postgres.<ref>:<pw>@aws-0-<region>.pooler.supabase.com:5432/postgres` |

The pooler doesn't support logical replication, so we can't use it for the
worker — but it's dual-stack (IPv4 + IPv6), so it works fine for the test's
simple INSERTs from your laptop.

## 6. Run the test

```sh
# --- Rawtree credentials ---
export RAWTREE_API_KEY="rt_..."
export RAWTREE_ORG="your-org"
export RAWTREE_PROJECT="your-project"
# Production host — never use api.us-east-1.aws.rawtree.dev (internal):
export RAWTREE_URL="https://api.us-east-1.aws.rawtree.com"

# --- Supabase ---
# Worker uses the direct (IPv6) URL — required:
export RAWTREE_TEST_SUPABASE_DATABASE_URL='postgres://postgres:PASSWORD@db.<ref>.supabase.co:5432/postgres?sslmode=require'
# Test inserts from your laptop go through the pooler (dual-stack) — optional but
# typically required because home/office networks don't have IPv6 routes:
export RAWTREE_TEST_SUPABASE_INSERT_URL='postgres://postgres.<ref>:PASSWORD@aws-0-us-east-1.pooler.supabase.com:5432/postgres'
# Supabase CA cert (required in practice; without it TLS verification fails):
export RAWTREE_TEST_SUPABASE_TLS_ROOT_CERT_PATH="$HOME/supabase-ca.crt"

# Optional: how long to wait for the inserted rows to reach Rawtree.
# Default 15m.
export RAWTREE_E2E_TIMEOUT=15m

# --- AWS ---
# Standard AWS credentials must be available (env vars, SSO, etc.). The test
# region is hard-coded to us-east-1.

TF_ACC=1 go test -v -timeout 30m \
  -run TestAccSupabaseCDCIngestion_endToEndData \
  ./internal/resources/supabase_cdc_ingestion/
```

What the test does, in order:

1. Pre-check: confirms env vars are set; creates the ECS service-linked role
   (`AWSServiceRoleForECS`) if missing.
2. Provisions a throwaway dual-stack VPC + IPv6 subnet + IGW via the AWS
   provider (so the Fargate task can reach Supabase over IPv6).
3. Creates the `rawtree_supabase_cdc_ingestion` resource — Secrets Manager
   secret, log group, IAM role, ECS cluster, task definition, and the
   long-running Fargate service (one task, `desiredCount=1`).
4. Reads the current row count in the Rawtree destination table
   (`public_superstore__sales__data`).
5. Inserts 5 synthetic rows into `public.superstore_sales_data` via the
   pooler URL, using random `Row ID`s in the billions to avoid PK conflicts.
6. Polls Rawtree's query API until the count grows by ≥ 5 (or
   `RAWTREE_E2E_TIMEOUT` expires).
7. Destroys everything — VPC, ECS resources, IAM role, log group, secret.
   The Postgres replication slot is **not** dropped; see "Cleanup" below.

## 7. Cleanup between runs

The test leaves one thing behind in your Supabase database: the replication
slot `supabase_etl_apply_1`. That's by design — successive test runs reuse it,
so the worker resumes from the prior LSN instead of doing a full table copy
every time. The slot pins WAL though, so if your Supabase project sits idle
for a long time and you want to free that WAL, drop it:

```sql
SELECT pg_drop_replication_slot('supabase_etl_apply_1')
WHERE EXISTS (
  SELECT 1 FROM pg_replication_slots
  WHERE slot_name = 'supabase_etl_apply_1' AND NOT active
);
```

The synthetic test rows accumulate in `public.superstore_sales_data` (5 per
run). To remove them:

```sql
DELETE FROM public.superstore_sales_data
WHERE "Order ID" LIKE 'TEST-%';
```

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| `Failed to run initialization ECS task` | Init was meant to be off; if you've edited the config, set `run_initialization_task = false`. |
| `Network is unreachable` in worker logs | VPC didn't get an IPv6 path. Check the route table includes `::/0 → IGW` and the subnet has an IPv6 CIDR. |
| `InvalidCertificate(UnknownIssuer)` | Missing `RAWTREE_TEST_SUPABASE_TLS_ROOT_CERT_PATH` or the file doesn't contain the Supabase CA. |
| `RawTree ingest failed: status=502 Bad Gateway` | `RAWTREE_URL` points at `.dev` or another env where your account doesn't exist. Use `https://api.us-east-1.aws.rawtree.com`. |
| `connect: no route to host` from the test process | The test is connecting to Supabase direct from your laptop. Set `RAWTREE_TEST_SUPABASE_INSERT_URL` to the pooler URL. |
| `Failed to inspect source database recovery state` | The worker can reach Postgres at the TCP level but is failing on something at handshake. Check the CA path, `sslmode` in the URL, and that the user has `REPLICATION` role. |
| `table sync worker failed ... DestinationQueryFailed` | Worker reached Rawtree's API but got a non-2xx. Most often a 502 from a wrong `RAWTREE_URL`. |
